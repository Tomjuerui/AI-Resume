# 智能简历分析系统 (AI-Resume-Analyzer) PRD

### 1. 产品概述

本产品是一个基于 Serverless 架构的轻量级、智能化的简历筛选后端服务及配套交互门户。它旨在通过 AI 大语言模型（LLM）的自然语言理解能力，自动化解析非结构化的 PDF 简历，精准提取候选人核心特征，并将其与结构化的岗位需求（JD）进行智能匹配与双向评分，从而将 HR 或技术面试官从海量的人工“初筛”工作中解放出来。

### 2. 目标用户

- **HR / 招聘专员：** 需要快速对收到的海量简历进行初步意向过滤。
- **技术面试官 / 用人部门负责人：** 在面试前需要快速掌握候选人的技能匹配度（技能树雷达图概念）和项目经验亮点。
- **求职者（潜在）：** 未来可扩展为候选人自测简历匹配度的工具。

### 3. 核心功能

- **上传与基础解析：** 支持单文件 PDF 格式简历上传，多页文本精准提取与降噪处理（清洗乱码、多余换行）。
- **AI 智能信息抽取：** 将清洗后的非结构化文本转化为强类型 JSON。提取基础信息（姓名、联系方式）及深度信息（求职意向、薪资、年限、学历、项目亮点）。
- **JD 语义匹配与评分引擎：** 接收岗位需求文本，利用 AI 进行双向推理计算，输出综合匹配度得分（0-100分），并附带简短的“匹配理由”与“风险提示”。
- **缓存加速机制：** 对已解析的简历文件（基于文件 Hash/MD5）和匹配结果进行 Redis 缓存，实现相同文件的秒级响应。
- **极简交互大盘：** 一个直观的 Web 页面，左侧上传+填 JD，右侧实时渲染评分结果与解析 JSON。

### 4. 功能优先级 (MVP 策略)

在 24 小时的极限开发下，严格执行优先级切割是成功的关键。

- **P0（核心链路，必须在首个 8 小时内打通）：**
  - RESTful API: `/api/upload` (单 PDF 上传及基础文本提取)。
  - API: `/api/analyze` (调用大模型提取**基础必选字段**：姓名、电话、邮箱、地址)。
  - 简易匹配：JD 关键词基于规则或简单 Prompt 的匹配评分。
  - 使用 AI 工具快速生成前端基础骨架。
- **P1（高价值功能，后续 10 小时攻坚）：**
  - AI 深度信息抽取（教育背景、项目经历等加分项）。
  - AI 深度语义评分（替代简单的关键词匹配，让模型给出多维度的匹配报告）。
  
  
- **P2（加分项与优化，最后 6 小时打磨）：**
  - 友好的前端加载动画（SSE 伪流式输出或 Skeleton 骨架屏）。
  - 结构化日志与完善的错误重试机制。

### 5. 界面设计建议

考虑到前端允许全量 AI 辅助，推荐采用“双栏对比交互（Split View）”设计，技术感强且效率高。

- **整体布局：** 左右分栏。
- **左侧操作区（Input）：**
  - 顶部：拖拽式的文件上传组件（仅限 PDF）。
  - 底部：一个大的文本域（Textarea），用于粘贴岗位需求描述（JD）。
  - 底部居中：一个显眼的“AI 智能分析”按钮（带 Loading 状态）。
- **右侧结果区（Output）：**
  - 顶部卡片：核心看板（候选人姓名 + 圆形进度条展示 **匹配度分数**）。
  - 中部卡片：雷达图或标签云展示“匹配亮点”与“缺失技能”。
  - 底部折叠面板（Accordion）：展示 AI 抽取的完整 JSON 结构数据，方便开发者/验收者查看底层结果。

### 6. 技术栈建议

结合 Serverless 环境和极速开发的需求，使用以下“现代化”技术栈：

- **后端开发语言：** Python 3.10+
- **后端框架：** FastAPI。
- **AI 交互 / Agent 逻辑：** 推荐直接使用大模型提供商的官方 SDK，或者轻量级的 Prompt 模板。若想展现 Agentic 思想，可以构建简单的串行工作流（Router -> Extractor Agent -> Scorer Agent）。
- **PDF 解析库：** `pdfplumber`（对文字排版提取较好）或 `PyMuPDF`（速度极快）。
- **缓存：** Redis（如无外置组件条件，演示时可短暂降级为内存 LRU Cache，但在设计思路上必须预留接口）。
- **前端工具：** 强烈建议使用 Vercel v0 或 Bolt.new，通过一句话 Prompt（"A split-screen resume analyzer dashboard with Next.js, Tailwind CSS, and Shadcn UI"）直接生成 React 页面代码。

### 7. 代码风格与架构模式

为了保证系统的高可维护性，建议采用经典的**分层架构**：

- **Controller 层 (`routers/`)：** 仅负责接收 HTTP 请求、参数校验、调用 Service 层并返回标准 JSON 结构。
- **Service 层 (`services/`)：**
  - `ResumeParserService`：负责 PDF 读取和文本降噪。
  - `AIAgentService`：封装 LLM 的调用逻辑。建议将 Prompt 集中管理，采用结构化输出（如使用 OpenAI 的 `response_format={ "type": "json_object" }`）以确保返回格式的稳定性。
  - `MatchEngineService`：处理具体的评分策略（计算权重、整合特征）。
- **基础设施层 (`infra/`)：** 封装 Redis 连接、阿里云 OSS/FC 相关的客户端。
- **数据校验 (`schemas/`)：** 统一定义 Pydantic BaseModel，确保出入参类型安全。

### 8. 限制条件与边界场景 (Corner Cases)

优秀的系统设计必须直面异常。在实战或代码评审中，对这些场景的处理往往是拉开差距的关键点：

1. **文件异常拦截：**
   - 非 PDF 文件或恶意伪装后缀拦截。
   - 文件体积过大（例如 > 10MB）拒绝服务。
2. **解析失败（OCR 陷阱）：**
   - 纯图片类 PDF（如扫描件）：常规 PDF 库无法读取文本。*处理策略：捕获文本提取为空的异常，返回特定 Error Code，提示“不支持扫描件，请提供纯文本 PDF”，避免无意义地调用大模型。*
3. **大模型“幻觉”与 JSON 格式化失败：**
   - 即使要求输出 JSON，大模型偶尔也会在首尾附加 Markdown 标记（如 ````json`）。
   - *处理策略：* 在代码中增加后置清洗正则，或者引入重试机制（Retry），若解析强类型对象失败，则最多重试 2 次。
4. **冷启动与 Serverless 超时：**
   - 大模型生成长文本可能耗时 10-20 秒。阿里云 FC 有执行超时限制（API Gateway 通常限制在 30s-60s）。
   - *处理策略：* 精简 Prompt，要求大模型输出极简 JSON；前端必须设置合理的 Timeout 时间，并提供安抚性质的加载动画。
5. **敏感信息保护：**
   - 系统仅做瞬时解析，内存中流转，日志中坚决脱敏（隐藏真实姓名、手机号），响应完成后尽快清理本地临时存储（如 `/tmp` 目录）。
